# async-rl-lab

`async-rl-lab` is a small educational and research-oriented async RL lab for LLM and agentic systems.

The goal is simple: make async RL systems easier to reason about.

Instead of hiding rollout generation, reward verification, policy publication, inference serving, and learner updates behind one trainer call, this repo keeps those pieces explicit and observable. It is designed for people who want to understand where policy lag comes from, how grouped rollouts behave under asynchronous learning, and why delayed verification changes training dynamics.

## What This Repo Is For

This repo is for learning and experimentation, not for benchmarking or production training.

It helps you study:

| Concept | What you can see in this repo |
| --- | --- |
| Policy lag | Every trajectory records a behavior-policy version and the learner measures lag when it consumes samples. |
| Async policy adoption | Publication and adoption are separate events, with actor-specific delay, jitter, and partial rollout. |
| Grouped rollouts | GRPO-style prompt groups stay intact through generation, buffering, and learner batching. |
| Delayed verification | Rewards can arrive later through a verifier pipeline instead of inline in the actor loop. |
| Inference serving | Generation is treated as a serving subsystem with batching, request schemas, refresh latency, and policy identity. |
| Tool-use environments | Multi-turn agent traces can include tool calls, tool results, and delayed reward. |
| Turn-aware objectives | The objective keeps turn structure, token masks, and answer/tool role masks instead of only flat sequence summaries. |

If you are new to async RL, the main educational value is that the code makes timing and provenance visible. You can inspect when a rollout was generated, which policy produced it, when reward arrived, when it entered the queue, and which learner version eventually consumed it.

## How The Pieces Fit Together

At a high level, the runtime looks like this:

1. Actors request generations from an inference engine using an explicit `GenerationRequest`.
2. The environment turns generated text into structured actions and transitions.
3. A verifier may reward trajectories immediately or later through a separate async path.
4. Complete rollout groups enter the grouped buffer.
5. The learner samples full groups, rescoring them under the current policy and a reference policy.
6. The learner publishes a new policy version.
7. Actors and inference engines adopt that policy later, not necessarily at the same time.

That separation is the main teaching device of the repo. It lets you see where stale data comes from and how asynchronous systems naturally create mixed-policy training batches.

## Core Ideas To Keep In Mind

| Idea | Why it matters here |
| --- | --- |
| Behavior policy vs learner policy | A rollout may be generated by an older policy than the one currently training. |
| Publish vs adopt | A learner can publish a new policy before the whole actor or serving fleet adopts it. |
| Group identity | GRPO-family methods depend on preserving prompt-group structure exactly. |
| Verification latency | Reward delay can dominate training freshness even when generation is fast. |
| Turn structure | Agentic traces are often easier to reason about per turn than as one flat string. |

## Quickstart

This repo uses `uv` for everything.

### Install and run the demo

```powershell
uv sync
uv run async-rl-demo
```

The demo runs a small async setup with:

- multiple actors
- grouped rollouts
- delayed verification
- async policy adoption
- a staleness-aware GRPO-style learner

It writes structured events under `artifacts/`.

### Run tests

```powershell
uv run python -m unittest discover -s tests -v
```

### Run dev checks

```powershell
uv sync --group dev
uv run --group dev ruff check .
uv run --group dev mypy src
```

### Optional Hugging Face backend

```powershell
uv sync --extra hf
```

The HF backend is intentionally minimal. It exists to make request batching, refresh latency, and token logprob extraction operational, not to provide a full production serving stack.

## A Good First Reading Order

If you are new to the repo, this order usually works well:

1. Read `src/async_rl_lab/demo.py` to see the smallest complete runtime.
2. Read `src/async_rl_lab/models.py` to understand the core records.
3. Read `src/async_rl_lab/runtime.py` to follow actor, learner, verifier, and adoption flow.
4. Read `src/async_rl_lab/inference.py` and `src/async_rl_lab/buffer.py` to understand async serving and rollout freshness.
5. Read `src/async_rl_lab/objectives.py` to see how grouped, turn-aware GRPO-style losses are assembled.

## What You Can Learn From This Repo

This lab is useful if you want to answer questions like:

- What does policy staleness look like in actual data structures?
- How do grouped rollouts survive asynchronous queues without losing prompt identity?
- What changes when reward arrives later than generation?
- Why is inference better modeled as a serving subsystem than a helper function?
- How should policy publication differ from policy adoption?
- What is the smallest setup where GRPO-family post-training starts to feel different from classic synchronous RL?
